# TASK-24: jadx Decompilation of classes2.dex -- CRC64 ACW Analysis

## Architecture: Xamarin Android Callable Wrappers (ACW)

Every CRC64 class follows an identical structural pattern -- they are thin Java wrappers
generated by the Xamarin/.NET MAUI build system. The actual application logic resides
entirely in the .NET/Mono runtime, accessed via JNI native method calls.

### ACW Pattern (universal across all 678 classes)

```java
package crc64XXXXXXXXXXXXXXXX;

public class SomeActivity extends BaseActivity implements IGCUserPeer {
    // 1. Method registration string (declares which Java methods bridge to .NET)
    public static final String __md_methods = "n_onCreate:...\\nn_onResume:...\\n";

    // 2. GC reference tracking for cross-runtime bridging
    private ArrayList refList;

    // 3. Native method declarations (JNI -> Mono runtime)
    private native void n_onCreate(Bundle bundle);
    private native void n_onResume();

    // 4. Static initializer registers .NET type with Mono runtime
    static {
        Runtime.register("Full.Dotnet.TypeName, AssemblyName", SomeActivity.class, __md_methods);
    }

    // 5. Constructor activates .NET type via TypeManager
    public SomeActivity() {
        if (getClass() == SomeActivity.class) {
            TypeManager.Activate("Full.Dotnet.TypeName, AssemblyName", "", this, new Object[0]);
        }
    }

    // 6. Each overridden method delegates to native (Mono) counterpart
    @Override
    public void onCreate(Bundle bundle) { n_onCreate(bundle); }

    // 7. IGCUserPeer: prevents .NET GC from collecting objects still referenced from Java
    public void monodroidAddReference(Object obj) { ... }
    public void monodroidClearReferences() { ... }
}
```

**Key insight**: The `__md_methods` string and `Runtime.register()` call together define the
complete Java-to-.NET method binding surface. No Java-side logic exists beyond delegation.
All behavioral analysis must target the decompiled C# code in `decompiled_csharp/`.

## Activity Lifecycle Override Patterns

The `__md_methods` string reveals exactly which lifecycle callbacks each activity overrides
in .NET code. From examining the full set:

**Common lifecycle overrides across most activities**:
- `onCreate` -- universal (every activity overrides this)
- `onResume` / `onPause` -- common for state refresh
- `onDestroy` -- common for cleanup
- `onSaveInstanceState` -- used by stateful activities (PIN login, main app)

**Data-passing overrides** (intent-based navigation):
- `onActivityResult(int, int, Intent)` -- used by **MainActivity**, **IntermediateActivity**, **MessageThreadFragment**
- `onNewIntent(Intent)` -- used by **WebAuthenticationCallbackActivity**, **WebAuthenticatorIntermediateActivity**

**Important limitation**: Because all lifecycle methods immediately delegate to
`n_methodName()` (JNI calls to Mono), the Java layer reveals **which** methods are
overridden but not **what data** is passed via intents. Intent extras, bundle keys, and
data serialization are all handled in the .NET layer. The Java ACW is truly just a
pass-through.

### WebView Usage Contexts

1. **AulaPortalWebView** + **AulaPortalWebViewFragment** + **WidgetWebViewFragment**: Used for
   embedding "portal widgets" -- third-party modules loaded as web content within the native app.
   The `AulaWebViewClient` intercepts HTTP auth requests (for SSO token injection) and overrides
   URL loading (to handle deep links / navigation from web content to native screens).

2. **AulaTextWebView**: Used for rendering rich text content (messages, posts). Overrides
   `onTouchEvent` to handle custom touch interactions within HTML content.

3. **DataPolicyWebView**: Displays the data/privacy policy. The client intercepts URL loading
   and page finish events.

4. **SearchHighlightedWebView**: Renders search results with highlighted matches. No method
   overrides at the Java level.

### JavaScript Bridge Assessment

**No `@JavascriptInterface` annotations or `addJavascriptInterface()` calls were found in
any decompiled Java class.** This means:

- There is no Java-side JavaScript bridge exposed to WebView content
- If JavaScript bridges exist, they are configured entirely in .NET code (via Xamarin's
  `WebView.AddJavascriptInterface()` binding or `evaluateJavascript()` calls)
- The `AulaWebViewClient.onReceivedHttpAuthRequest` override suggests authentication
  tokens are injected at the HTTP level rather than via JavaScript bridges
- URL interception via `shouldOverrideUrlLoading` is the primary communication channel
  from web content back to native code

## Firebase Messaging Service Analysis

### Java ACW Layer

`AulaFirebaseMessagingService` extends `FirebaseMessagingService` and registers two native
method overrides:
- `onMessageReceived(RemoteMessage)` -- called when a push notification arrives
- `onNewToken(String)` -- called when FCM registration token is refreshed

Both delegate immediately to .NET via JNI.

### .NET Implementation (cross-referenced from decompiled_csharp/)

The C# `AulaFirebaseMessagingService` reveals the actual logic:

```
OnMessageReceived(RemoteMessage):
  1. Calls NotifyNotificationReceived(message)
     - Extracts notification body text
     - Parses message.Data via FirebaseNotificationParser.Parse()
       (extracts "elementId" and "type" string keys from the data payload)
     - Logs: "Notification received: ElementId=X, Type=Y, Body=Z"
     - Sends parsed RemoteNotification to PushNotificationMessagingCenter
       (Xamarin.Forms MessagingCenter event bus)
  2. Calls UpdateAllNotificationsInApp()
     - Fires NotificationMessagingCenter.NotifyNotificationListChanged
       with NotificationChangeType=1, NotificationType=2

OnNewToken(String):
  - Empty implementation (token refresh is not handled)
```

### Push Notification Data Model

Firebase data payload keys:
- `elementId` -- identifies the notification target (message thread ID, post ID, etc.)
- `type` -- notification category string

These are parsed into a `RemoteNotification` object and distributed via the
`PushNotificationMessagingCenter` event bus, which triggers in-app notification updates
and navigation.

**Notable**: `OnNewToken` is empty, meaning the app does not re-register its FCM token
when it rotates. The token is presumably registered once during login and stored server-side.

## MainApplication: App-Level Lifecycle Tracking

`MainApplication` (the Android Application class) implements:
- `Application.ActivityLifecycleCallbacks` -- monitors all activity lifecycle transitions
  (created, started, resumed, paused, stopped, destroyed) with both pre/post variants
- `DefaultLifecycleObserver` -- Android architecture component lifecycle observation

This provides the .NET layer with full visibility into which activities are in the
foreground, enabling features like:
- Determining if the app is in the foreground when notifications arrive
- Tracking screen time / analytics
- Managing cross-activity state

## Key Findings

1. **ACWs are pure pass-throughs**: All 678 CRC64 classes contain zero application logic.
   Every method body is a single JNI call to the Mono runtime. The Java decompilation is
   valuable primarily for mapping the class hierarchy, identifying which Android callbacks
   are overridden, and confirming the WebView/Service/Receiver surface area.

2. **No JavaScript bridges in Java**: WebView communication happens via URL interception
   (`shouldOverrideUrlLoading`) and HTTP auth request handling, not JavaScript interface
   injection. Any JS bridge would need to be configured in .NET code.

3. **Firebase notifications use simple key-value data**: Only `elementId` and `type` strings.
   Token refresh is not handled (empty `OnNewToken`).

4. **Activity navigation is entirely .NET-managed**: The Java layer reveals the activity
   graph structure but not the intent extras or navigation parameters. For that, the
   decompiled C# in `decompiled_csharp/AulaNative.Droid/` must be analyzed.

5. **Portal widget WebViews handle HTTP auth**: The `AulaWebViewClient.onReceivedHttpAuthRequest`
   override indicates that portal widgets may require authentication credentials injected
   at the HTTP layer, suggesting SSO token forwarding to embedded web modules.

6. **Extensive lifecycle monitoring**: `MainApplication` tracks all activity lifecycle events
   with pre/post hooks, giving the .NET layer complete visibility into the app's foreground state.
